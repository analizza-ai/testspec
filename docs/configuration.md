# Configuration

## testspec.config.json

Generated by `testspec init` in the root of your project.

```json
{
  "sdd": "openspec",
  "agent": "claude",
  "stubs": { "unit": true, "integration": true },
  "unitFramework": "vitest",
  "qaRepo": "owner/repo",
  "loadHints": true,
  "chaosHints": true
}
```

| Field | Values | Description |
|---|---|---|
| `sdd` | `openspec` | SDD framework in use |
| `agent` | `claude` \| `copilot` | AI agent for test generation |
| `stubs.unit` | `true/false` | Generate unit test stubs |
| `stubs.integration` | `true/false` | Generate Testcontainers stubs |
| `unitFramework` | `vitest` \| `jest` \| `pytest` \| `junit` | Unit test framework |
| `qaRepo` | `owner/repo` | GitHub repo for tests.md handoff to QA skills |
| `loadHints` | `true/false` | Include load test hints in tests.md |
| `chaosHints` | `true/false` | Include chaos engineering hints in tests.md |

## Installed Claude commands

When `agent = "claude"`, `testspec init` writes 4 files to `.claude/commands/`:

| File | Slash command | Layer |
|---|---|---|
| `.claude/commands/testspec.md` | `/testspec` | Developer |
| `.claude/commands/testspec-specify-qa.md` | `/testspec-specify-qa` | QA |
| `.claude/commands/testspec-apply-qa.md` | `/testspec-apply-qa` | QA |
| `.claude/commands/testspec-run-qa.md` | `/testspec-run-qa` | QA |

## `testspec/` folder (QA project)

The QA skills create and maintain a `testspec/` folder in the **QA repo** (not the app repo):

```
testspec/
├── instructions.md              # app_repo, MCP config, project conventions
├── current-feature.md           # session lock — active feature name
└── {feature-name}/
    ├── spec.qa.md               # technical QA contract
    └── tasks.qa.md              # [ ] / [x] checklist of scripts to implement
```

### `testspec/instructions.md`

Created automatically by `/testspec-specify-qa` on first run. Edit it to configure MCPs and project defaults:

```markdown
# Instructions — QA Project

## app_repo
owner/repo

## Ferramenta de teste padrão
k6

## Arquitetura
- Scripts organizados por tipo: e2e/, load/, chaos-engineering/
- Um script por caso de teste (CT)
- Run plans .md gerados ao lado de cada script

## Tecnologias
- k6 (JavaScript) para E2E e load
- kubectl para coleta de logs k8s
- Confluence para publicação de relatórios

## Convenções
- Nomenclatura kebab-case sem acentos
- Variáveis de ambiente para URLs e credenciais (nunca hardcode)
- Thresholds: p95 < 500ms, error rate < 1% (load); p95 < 2000ms, error rate < 5% (chaos)
```

### `testspec/current-feature.md`

A session lock file. Once written, all 3 QA skills (`/testspec-specify-qa`, `/testspec-apply-qa`, `/testspec-run-qa`) read the active feature from it without asking again.

Content is just the feature name:
```
kafka-consumer-order-request
```

Clear it to switch features between sessions.

## QA test output structure

```
src/test/features/
└── {feature-name}/
    ├── e2e/
    │   ├── {tool}-e2e-{action}-{scenario}.js
    │   └── {tool}-e2e-{action}-{scenario}.md     ← run plan
    ├── load/
    │   ├── {tool}-load-{action}-{scenario}-{rps}-rps-{dur}.js
    │   └── {tool}-load-{action}-{scenario}-{rps}-rps-{dur}.md
    └── chaos-engineering/
        ├── {tool}-dr-{action}-{scenario}-{chaos-type}.js
        └── {tool}-dr-{action}-{scenario}-{chaos-type}.md

reports/
└── {feature-name}/
    └── {script}-{YYYY-MM-DD-HHmm}.md             ← fallback when Confluence unavailable
```

## MCP requirements

| Skill | Required MCPs |
|---|---|
| `/testspec-specify-qa` | GitHub (for `tests.md` fetch) |
| `/testspec-run-qa` | GitHub, kubectl/k8s, Splunk (optional), Confluence (optional) |

Configure MCPs in `testspec/instructions.md` and in `.mcp.json` at the project root.
